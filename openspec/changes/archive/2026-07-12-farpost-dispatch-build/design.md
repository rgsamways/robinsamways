## Context

Farpost Dispatch started as an unscoped lightbulb (`rsw-lb-farpost-dispatch`, logged 2026-07-10), written to close a real gap: no piece yet demonstrates building *inside* Salesforce (Experience Cloud, Apex, a portal a real external user logs into) rather than integrating with it from outside, the way Credential Flow does. Two open questions blocked scoping: Experience Cloud licensing on a free Developer Edition org, and the object model. Both are now resolved — Robin confirmed real, free, unused Partner Community licenses directly in his org (2026-07-11), and the object model conversation settled on AI-assisted matching as the technical centerpiece, with professionals self-claiming jobs rather than ops assigning them.

A third question — how this relates to Farpost's real, live dispatch engine — surfaced during scoping and matters enough to state as a load-bearing design constraint, not just a proposal footnote: the real product's dispatch system (`f13-dispatch-rebuild`, then `unify-claim-dispatch-on-work-request`) is mature and already generalized for reuse. This change has zero relationship to it beyond sharing a name and a founding story.

## Goals / Non-Goals

**Goals:**
- Ship a real, deployed Salesforce app proving genuine Experience Cloud + Apex + AI-callout-from-inside-Salesforce competency, not a Setup-UI screenshot tour.
- Make the AI matching feature a real, complementary counterpart to Credential Flow's existing Anthropic-powered recommendation — same "explain the why" pattern, opposite direction of integration (Apex calling out, not Python calling in).
- Directly echo Farpost's founding story (the rural service-availability gap) without touching, duplicating, or depending on any part of the real farpost.ca product.
- Make full, genuine use of the confirmed Partner Community licenses — professionals are real Partner Community users, not just profile records ops looks at.

**Non-Goals:**
- Any integration, data sync, or shared infrastructure with the real farpost.ca dispatch engine — this is a fully separate Salesforce org with no relationship to Farpost's MongoDB/FastAPI backend.
- Reusing Farpost Atlas's real StatCan geo boundary data — Coverage Area stays Salesforce-only, a simple picklist, not a custom object with geometry (per Robin's explicit call). No cross-piece data sharing.
- A live public embed or login link on robinsamways.ca — a free Developer Edition org's Partner Community login exposed publicly risks abuse and governor-limit exhaustion for no real benefit; a case study plus a `SetupGallery` of real screenshots tells the story just as well without that risk.
- A Guest User public job board — would add real scope (a second, unauthenticated access path with its own security surface) without serving this change's actual goal, which is proving the *partner-login* Experience Cloud pattern specifically.
- PostGIS-style spatial matching — region matching is a simple picklist-equality filter, not a geospatial join; that competency is already proven by Farpost Atlas.
- Automating the fully manual parts of Salesforce setup (Experience Cloud site creation via wizard, license assignment, Named Credential secret entry, Experience Builder page layout) — no local Salesforce CLI/org connection exists in this environment; these remain Robin's documented manual steps, same precedent as Farpost Pulse's Azure deployment.

## Decisions

- **A real Salesforce DX project (`pieces/farpost-dispatch-sf/`), not a documentation-only Setup-UI walkthrough.** Considered and rejected: treating this purely as manual configuration Robin does himself with no committed source, mirroring Credential Flow's original Connected App setup. Rejected because this portfolio's whole ethos is git-tracked, spec-driven, reviewable code — an entire piece built with nothing in version control would be a worse demonstration of engineering discipline than the thing it's trying to prove. Apex classes, custom object/field metadata, permission sets, and LWC bundles are all real, hand-authored source files checked into this repo, deployable via `sf project deploy start` once Robin authenticates the Salesforce CLI to his org.
- **Apex, satisfying "Portfolio piece isolation"'s runtime-difference trigger directly.** Apex only runs inside Salesforce — there's no question of whether this belongs in `api/`'s shared Python stack. `pieces/farpost-dispatch-sf/` is the second real instance of this convention's trigger #1 (a genuinely different runtime), after Farpost Pulse's Node.js.
- **Coverage Area is a Salesforce-only picklist field, not a custom object, and never touches Farpost Atlas's real geo data.** Robin's explicit call. A rural-region picklist (drawn from real North Hastings-area municipality names, for narrative continuity with Atlas's own setting, but zero data or code sharing) is sufficient for a categorical region-match filter — a full object with geometry would be over-engineering for what this matching logic actually needs, and duplicating Atlas's real StatCan-derived boundary data into a second, unrelated system would be exactly the kind of cross-piece coupling "Portfolio piece isolation" exists to prevent.
- **Professionals self-claim jobs; the AI ranks and surfaces, it doesn't assign.** Robin's explicit call (see proposal history). The AI matching service ranks eligible Professionals for a Job with natural-language reasoning, visible to ops on the Job record *and* to Professionals as a "recommended for you" sort on their own portal view — but claiming itself is a Professional action (`JobClaimService.claimJob`), first-claim-wins. This is the sharper version of the founding story: instead of manual outreach that failed, the system proactively surfaces the job to the best-fit people, who claim it themselves.
- **Anthropic callout originates from Apex via a Named Credential, not from a shared `api/` proxy.** Considered and rejected: routing the AI call through `api/app/ai.py` (already handles Anthropic calls for Credential Flow) to avoid a second integration. Rejected because the entire point of this piece is proving the AI call can be made *from inside* Salesforce — routing it through `api/` would just be Credential Flow's pattern again with extra steps, not a distinct skill. A Named Credential (pointing at `https://api.anthropic.com`, with the real API key entered manually by Robin as the External Credential's principal — never committed, same secrets-not-in-git precedent as every other API key in this project) is the current recommended Salesforce pattern for a secure external callout, avoiding the common anti-pattern of storing a key in a Custom Metadata Type field in plaintext.
- **Claim concurrency handled with `SELECT ... FOR UPDATE`, not last-write-wins.** Two professionals could plausibly claim the same job near-simultaneously in a demo click-through. `JobClaimService.claimJob` row-locks the `Job__c` record, re-checks `Status__c == 'Open'` inside the lock, and only then updates — a genuinely correct concurrency pattern worth pointing to in the case study, not just a toy.
- **Certifications and regions are picklists (multi-select for Certifications on Contact, single-select for Region on both Contact and Job__c), not child objects.** At this scale (a handful of fictional professionals and jobs) a full junction-object model would add real complexity without adding real matching fidelity — picklist-equality filtering is sufficient to demonstrate the matching logic, and keeps the object model legible in a case study.
- **No live demo widget on robinsamways.ca; a case study plus a deferred `SetupGallery` instead.** Every other Farpost-hub piece (Atlas, Pulse) and Credential Flow embed a live, safely-scoped interactive demo. Dispatch's real interactive surface is a login-walled Partner Community portal — there is no safe way to expose that publicly without either sharing real credentials (inviting abuse of a free-tier org with hard governor limits) or building a second, unauthenticated access path (a Non-Goal above). Real screenshots of the actual portal (professional's job list, the claim action, the AI recommendation panel on the ops side) tell the story without that risk, following the exact `SETUP_GALLERY` precedent `CLAUDE.md` already establishes — deferred until Robin has actually done the configuration, same as Atlas's and Pulse's own still-pending galleries.

## Risks / Trade-offs

- [No local Salesforce runtime exists in this build environment — CLI cannot deploy, run, or independently verify any Apex code or test class before handoff] → the most significant departure from every prior piece's verification bar. Mitigated by writing real, complete Apex test classes (with `HttpCalloutMock` for the Anthropic callout, so tests never make live API calls) that Robin runs himself via `sf apex run test` once deployed — documented explicitly as his verification step, not silently assumed to pass.
- [Free Developer Edition org governor limits (callout limits, API call limits) could be hit if the matching service is triggered aggressively] → low real risk at this piece's fictional-data scale (a handful of professionals/jobs, callouts only on-demand from the recommendation panel, not on every page load), but worth a deliberate note in the Apex service rather than an unbounded loop.
- [Building Experience Builder page layouts is inherently point-and-click, not fully metadata-authored by hand] → accepted; the LWC components themselves are real committed source, but wiring them onto the page layout is Robin's manual step, same category as Atlas's Railway provisioning or Pulse's Azure Function App deployment.
- [A second, unrelated Salesforce app named "Dispatch" living alongside the real farpost.ca product risks reader confusion] → mitigated by stating the non-relationship explicitly and prominently in the case-study page's opening section, not just in this design doc.

## Migration Plan

1. Scaffold `pieces/farpost-dispatch-sf/` as a Salesforce DX project (`sfdx-project.json`, `force-app/main/default/`).
2. Author custom field metadata on Contact (Service_Region__c, Certifications__c, Availability_Status__c, Rating__c) and the `Job__c` custom object (Job_Type__c, Region__c, Urgency__c, Status__c, Assigned_Professional__c, Description__c).
3. Author `JobMatchingService.cls` (eligible-candidate query + Named Credential callout to Anthropic + ranked response parsing) and `JobClaimService.cls` (row-locked, status-guarded claim), plus their test classes with `HttpCalloutMock` coverage.
4. Author the Named Credential/External Credential metadata (definition only — the real API key is Robin's manual entry) and a `Farpost_Dispatch_Partner` permission set scoping Partner Community access.
5. Author the two LWC bundles: `jobRecommendationPanel` (ops-side, Job record page) and `openJobsBoard` (Partner Community portal page, with the Claim action).
6. Write an Apex data-seeding script (`scripts/apex/seed.apex`, run via `sf apex run` once deployed) creating fictional professionals and jobs spanning multiple regions/certifications/urgency levels with genuine matching variation.
7. Replace `web/src/app/farpost/farpost-dispatch/page.tsx`'s placeholder with the real case-study page; update the `farpost-page-content` spec delta.
8. Document Robin's manual deployment/configuration steps in `docs/deployment-guide.md`'s "Portfolio piece deployments" part and sync `/ops/deploy`; add new entries to `docs/stack.md`.
9. Robin installs the Salesforce CLI, authenticates to his Developer Edition org, deploys the DX project, runs the seed script, runs the Apex tests, creates the Experience Cloud site and assigns Partner Community licenses, enters the real Anthropic API key into the Named Credential, and builds the Experience Builder page layout — outside this change's own scope, documented for a clean handoff (same pattern as Pulse's Azure deployment and Atlas's Railway provisioning).
10. Once Robin has done step 9, a follow-up captures real setup screenshots and builds the `SetupGallery` — a separate, non-blocking task logged via the handoff-logging convention, not part of this change.
